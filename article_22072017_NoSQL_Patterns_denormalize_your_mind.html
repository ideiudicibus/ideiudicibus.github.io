The Relational Data approach.

In the RDBMS world that is relational databases, the data modeling processes typically starts by modeling data as a series of tables, consisting of rows and columns, which define the so called schema of your data. Relational database theory has defined  a number of ways of putting application data into tables, referred to as normal forms.  Therefore, understanding modeling related concepts is key to work on any data related project whether is SQL or NoSQL base.Here' I'll cover all of them with simple definitions.


  1. First Normal Form. This makes sure there is no data redundancy and a primary key is well defined on the table. For example, we are not capturing multiple values in a single column (such as comma separated) or using multiple columns instead of rows to capture data with varying fields/identifiers.
  2. Second Normal Form. This makes sure that all columns in the table have 1-1 direct relationship with the entire primary key of the table. Attributes/Columns that likely describe key column of the table can be stored separately. Foreign key is defined to be able to join the multiple tables generated as a result of this type of normalization.
  3. Third Normal Form.This makes sure that any columns describing the non key columns are stored separately and is a higher level of normalization.

The 3NF normalized form is the most of the times a SQL  nirvana. The nice thing about normalization is that it allows for easy  updating without any redundancy. Each fact about the application domain can be updated
by changing just one value, at one row-column intersection. The problem arises
when you try to get the data back out. For instance, in a phone book database, we  may want to have a form that displays a contact along with all of his or her phone
numbers. In cases like these, the relational database programmer reaches for a  classic JOIN. 

The real problem is in what the database had to do to create this result set, particularly if the
database is backed by a spinning magnetic disk. To see why, we need to briefly look at
some of the physical characteristics of such devices.
Spinning disks have the property that it takes much longer to seek to a particular location on the disk than it does, once there, to sequentially read data from the disk so that a modern disk can take up to 5 milliseconds to seek and reach the block location that is the "place" where it can begin reading. This can lead to a performance problem when record numbers arise and whe tables are made up of several attributes, that is waiting minutes for a join to performs. Now, some techniques have been evolved in order to mitigate the phenomenon, still nowadays a developer has to cope with the problem of de-normalization. So at the begining we started with theorically correct 3NF and then we'll ended up with the problem of flatting most attributes as possibile so that our joins performance stills remains good.
